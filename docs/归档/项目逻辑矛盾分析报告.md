# 项目逻辑矛盾分析报告

**版本**: 1.0  
**日期**: 2026-05-27  
**分析范围**: 目录结构、测试、数据来源、数据处理、指标体系、命令路由、配置文件

## 执行摘要

经过详细调研，发现当前项目存在 **10 个主要逻辑矛盾点**，涵盖数据缓存、测试基础设施、数据源优先级、评分体系一致性等方面。其中 **3 个为高优先级**（直接影响功能），**4 个为中优先级**（影响维护和一致性），**3 个为低优先级**（历史遗留或文档问题）。

## 详细矛盾分析

### 1. 数据缓存路径不一致 [高优先级]

**问题描述**:
- 项目中存在 **两个 prices.json 文件**：
  1. `data/prices.json`（根目录，仅 1 个条目）
  2. `src/stock_analysis/data/prices.json`（源码目录，完整数据）
- `data/companies.json` 包含 **26 家公司**
- AGENTS.md 声称 prices.json 包含 "26家公司缓存"，但未指明具体路径

**矛盾点**:
- 验证器 (`validate.py`) 使用 `engine_dir / "data" / "prices.json"`，指向 `src/stock_analysis/data/prices.json`
- AGENTS.md 中的目录结构图显示 `data/prices.json`，指向根目录
- Web App 后端可能读取错误的 prices.json 文件
- 用户可能混淆两个文件的位置

**影响**:
- 数据验证可能读取错误的文件
- 维护困难，需同步两个文件
- 文档与实际不符

### 2. 测试基础设施缺失 [高优先级]

**问题描述**:
- `tests/` 目录存在多个测试文件（test_batch.py, test_comprehensive.py 等）
- `pyproject.toml` 声明 pytest>=9.0 为开发依赖
- **pytest 未安装**，运行 `python3 -m pytest` 报错 "No module named pytest"

**矛盾点**:
- AGENTS.md 提供测试命令 `PYTHONPATH="src" pytest tests/ -v`
- 实际无法运行测试
- 测试文件存在但无法执行

**影响**:
- 无法验证代码正确性
- 无法进行回归测试
- 开发流程不完整

### 3. 数据源优先级与实现不符 [高优先级]

**问题描述**:
AGENTS.md 声明数据源优先级：
```
官方披露/公司 IR > 交易所/XBRL > 行情层 > 二手聚合站
```

**实际实现**:
- 主要使用 **yfinance**（行情层）获取财务数据
- 使用 **Google Finance**（二手聚合站）获取价格数据
- **未实现**官方披露/公司 IR 和交易所/XBRL 数据源

**矛盾点**:
- 文档声明的优先级与实际实现不符
- yfinance 作为行情层数据源，优先级应低于官方披露
- 缺少 XBRL 解析器（尽管文档提到 "不写 XBRL 解析器"）

**影响**:
- 数据质量可能不如官方披露
- 用户对数据来源有错误预期
- 无法满足对数据源有严格要求的用户

### 4. 评分体系版本混乱 [中优先级]

**问题描述**:
- AGENTS.md 声称使用 **v3.0 四层加权排名**
- schema.py 声称是 **Greenblatt 排名法 v3.0**
- greenblatt.py 仍包含 `score_10` 概念（1-10 十分制）

**矛盾点**:
- "排名优于打分" 是核心原则，但仍有打分系统
- `composite_to_score10` 函数将排名转换为打分
- `score_10` 字段在 RankingResult 中存在

**影响**:
- 与 "排名优于打分" 原则矛盾
- 可能混淆用户理解
- 维护两套系统增加复杂度

### 5. 命令路由不一致 [中优先级]

**问题描述**:
- AGENTS.md 声明命令为 `stock-analysis <公司中文名>`
- 实际测试使用 `PYTHONPATH="src" python3 -m stock_analysis.cli <公司中文名>`
- pyproject.toml 定义了 `stock-analysis` 入口点

**矛盾点**:
- 用户可能不知道需要设置 PYTHONPATH
- `stock-analysis` 命令可能未安装（需要 `pip install -e .`）
- 文档示例与实际使用方式不符

**影响**:
- 用户困惑
- 入门门槛高
- 文档可读性差

### 6. 加密资产数据源不完整 [中优先级]

**问题描述**:
- ETH staking: Etherscan V1 API 已废弃，需迁移到 V2
- BNB staking: BscScan API 需要免费 key，当前用占位
- CoinGecko 通胀率端点响应大，需优化

**矛盾点**:
- AGENTS.md 声称支持 4 种加密资产（BTC, ETH, SOL, BNB）
- 实际只有 SOL staking 数据完整（71.0%）
- ETH 和 BNB 数据不完整或缺失

**影响**:
- 加密资产排名不准确
- PoS 加密四层排名体系无法完整实施
- 用户体验不一致

### 7. 验证器依赖问题 [中优先级]

**问题描述**:
- `validate.py` 的 `validate_data_sanity` 函数依赖 prices.json 进行交叉验证
- prices.json 数据不完整（仅 1 个条目）
- 验证器可能因缺少基准数据而失败

**矛盾点**:
- 验证器设计为数据真实性检查工具
- 但缺少完整的基准数据
- 可能产生误报或漏报

**影响**:
- 数据验证不可靠
- 可能误导用户
- 无法保证报告质量

### 8. 目录结构与文档不完全一致 [低优先级]

**问题描述**:
- AGENTS.md 显示 `src/investskill/` 是 "方法论层 (上游开源, prompts + 21 skills)"
- 实际 `src/investskill/` 包含大量文档、插件、脚本等
- 与核心引擎 (`src/stock_analysis/`) 的关系不明确

**矛盾点**:
- investskill 目录结构复杂，包含 38 个条目
- 与核心引擎的集成方式不清晰
- 可能存在重复或冲突的代码

**影响**:
- 代码组织混乱
- 维护困难
- 新贡献者困惑

### 9. 历史遗留引用 [低优先级]

**问题描述**:
- `src/investskill/prompts/git-commit.md` 仍包含 VM Score 引用
- 其他文件中仍有 "Greenblatt 三层排名" 引用（如 schema.py 第 160 行）

**矛盾点**:
- 当前系统是四层加权排名，但仍有三层排名引用
- VM Score 是旧评分系统，已弃用但仍有引用

**影响**:
- 代码库不干净
- 可能误导开发者
- 历史包袱

### 10. 多个 TICKER_MAP 来源 [低优先级]

**问题描述**:
- `registry.py` 是单一数据源，提供所有映射
- `fetcher.py` 中仍有 `TICKER_MAP = name_zh_to_ticker()` 和 `_ALIASES` 硬编码
- `scaffold.py` 中也有 `_ALIASES` 硬编码

**矛盾点**:
- 单一数据源原则被部分违反
- 别名映射分散在多个文件中
- 维护时需同步多个位置

**影响**:
- 维护成本高
- 容易出现不一致
- 违反 DRY 原则

## 总结

项目存在 **10 个逻辑矛盾点**，其中 3 个高优先级问题直接影响功能。建议按优先级顺序修复：

### 立即修复（高优先级）
1. **统一数据缓存路径** - 删除根目录的 `data/prices.json` 或创建符号链接
2. **安装测试基础设施** - 运行 `pip install -e ".[dev]"` 安装 pytest
3. **对齐数据源优先级** - 更新 AGENTS.md 明确实际数据源

### 短期修复（中优先级）
4. **统一评分体系** - 移除 `score_10` 或明确其辅助地位
5. **统一命令路由** - 提供完整的安装和使用指南
6. **完善加密资产数据源** - 迁移 ETH staking 到 Etherscan V2
7. **修复验证器依赖** - 确保 prices.json 数据完整

### 长期优化（低优先级）
8. **整理目录结构** - 明确 investskill 与核心引擎的关系
9. **清理历史遗留引用** - 替换 VM Score 和三层排名引用
10. **统一 TICKER_MAP 来源** - 确保所有映射来自 registry.py

### 验证修复效果


修复后，项目将更加一致、可靠、易维护。所有矛盾点都将得到解决，用户将获得更好的体验。
